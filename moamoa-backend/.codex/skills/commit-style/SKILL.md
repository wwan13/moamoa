description: "moamoa 저장소의 커밋 메시지 규칙과 커밋 분할 기준을 최근 커밋 패턴에 맞춰 적용할 때 사용. staged diff 또는 작업 변경점을 보고 `type(scope) : summary` 형식 메시지를 제안하고, 커밋을 나눌지와 누락된 검증 포인트가 있는지 점검할 때 이 스킬을 사용."
---
moamoa 저장소의 최근 커밋 패턴을 기준으로 커밋 메시지와 커밋 단위를 정리한다.
- 커밋 메시지를 대신 써달라고 할 때
- staged diff를 보고 적절한 커밋 단위로 나눠달라고 할 때
- 작업 변경점이 `feat`인지 `fix`인지 `style`인지 모호할 때
- 커밋 전에 빠진 테스트나 라우트/API 연결을 체크하고 싶을 때
- 형식은 기본적으로 `type(scope) : summary`
- 최근 자주 쓰인 `type`: `feat`, `fix`, `style`, `chore`
- 최근 자주 쓰인 `scope`: `front`, `server`
- summary는 짧은 한글 구문으로 쓴다.
- 한 커밋에는 보통 하나의 문제 해결 또는 하나의 기능 추가만 담는다.
- `moamoa-web`, `moamoa-admin/admin-web` 변경은 기본적으로 `front`
- `moamoa-core`, `moamoa-support`, `moamoa-crawler`, `moamoa-admin/admin-api` 변경은 기본적으로 `server`
- `.gitignore`, 개발환경 설정, 배포/도구 설정은 `chore`를 우선 검토한다.
- 프론트와 서버가 함께 바뀌었지만 목적이 하나라면 분리 가능성을 먼저 검토한다.
- 분리할 수 없는 단일 작업이면 더 큰 영향 범위를 대표하는 scope 하나만 고른다.
- `feat`: 새 페이지, 새 API, 새 배치, 새 템플릿, 새 동작 추가
- `fix`: 버그 수정, 라우팅/쿼리/조건/HTTP method 오류 수정, 컴파일 오류 수정
- `style`: CSS, 레이아웃, 문구, 퍼블리싱처럼 동작 변화가 거의 없는 수정
- `chore`: 설정, ignore, 개발도구, 실행 환경, 의존성 정리
1. 먼저 변경 파일 목록을 보고 `front`/`server`/`infra` 성격을 분류한다.
2. 사용자 설명보다 실제 diff의 결과를 우선해 `type`을 고른다.
3. summary는 문제 또는 결과 하나만 드러내게 20자 안팎으로 압축한다.
4. 서로 독립적인 변경이 섞여 있으면 커밋 분리를 먼저 제안한다.
5. 최종 메시지는 후보 1개를 우선 제시하고, 애매하면 대안 1개만 추가한다.
- 프론트와 서버를 동시에 수정했더라도 배포/검증 포인트가 다르면 분리한다.
- 템플릿 추가와 배치 연결처럼 단계가 다른 서버 작업은 분리 가능성을 본다.
- CSS 수정과 기능 수정이 함께 있으면 가능한 한 분리한다.
- 테스트 수정만 별도라면 원인 커밋에 함께 포함하는 쪽을 우선 검토한다.
- 서버 기능 변경인데 테스트 또는 검증 경로가 전혀 보이지 않는가?
- 라우트 추가인데 `AppRoutes` 또는 엔트리 연결이 빠졌는가?
- API method 변경인데 프론트/서버 양쪽 호출 규약이 같이 맞춰졌는가?
- 배치/템플릿 변경인데 스케줄러, 설정, 테스트가 같이 갱신됐는가?
- `style`로 보이지만 실제 동작 조건을 바꾸는 코드가 섞여 있지 않은가?
- 기본 출력은 커밋 메시지 1개와 근거 1줄
- 커밋 분리가 필요하면 나눠야 하는 기준을 먼저 설명하고 메시지를 여러 개 제안
- 사용자가 원하면 `git commit`에 바로 넣을 수 있게 한 줄 메시지만 다시 출력
- `feat(front) : 공지사항 페이지 퍼블리싱`
- `fix(server) : 전체 게시글 조회 시 size 20을 다 채우지 못하는 오류`
- `style(front) : 카테고리 탭 액티브 색상 변경`
- `chore : air, superset 환경 설정`
